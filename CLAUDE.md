# CLAUDE.md — mcp-tools-server

Working notes for AI agents. Generated from a code-level survey (2026-07-23); verify paths still exist before relying on them.

Portfolio demo MCP server ("portfolio-tools") built on the official Python SDK's FastMCP. Exposes four deterministic, offline, least-privilege tools — search_web (keyword ranking over a curated 14-entry local index), query_database (single read-only SELECT against a lazily-seeded SQLite job-market demo DB, guarded by an sqlite3 authorizer), read_file / list_dir (filesystem sandboxed to DATA_DIR) — plus docs:// MCP resources for the files in data/docs. Runs over stdio (default, for Claude Desktop/Inspector) or streamable HTTP on :8082/mcp (Docker/compose; sibling repo agent-orchestrator calls search_web over HTTP).

## Commands
```bash
make install-dev && make test
make lint && make typecheck
make run-stdio                  # stdio transport (default)
make run-http                   # streamable HTTP on :8082/mcp
make lock
```

## Layout
- `app/server.py` — FastMCP wiring only: create_server(settings) registers four thin @server.tool() wrappers whose docstrings ARE the LLM-facing tool descriptions (they carry the DB schema, examples, constraints) and whose TypedDict return annotations become MCP output schemas / structuredContent. Also registers docs:// resources, selects transport (stdio default, --transport http => streamable-http; no SSE fallback — mcp>=1.10 always has streamable-http). Module-level `server = create_server()` exists so `mcp dev app/server.py` finds it.
- `app/tools/search.py` — search_web implementation: offline stub ranking data/search_index.json entries by keyword-overlap (keywords weight 2, title words weight 1; ties break on index order => fully deterministic). Caps: 2000-char query, 20 results. Raises ToolError for empty/tokenless query or missing/malformed index.
- `app/tools/database.py` — query_database implementation with 4-layer defence: (1) SQLite URI mode=ro connection, (2) authorizer allowing only SQLITE_SELECT/READ/FUNCTION/RECURSIVE and denying all else, (3) multi-statement rejection via sqlite3.complete_statement (';' inside literals is fine), (4) size caps: 20k-char SQL, 200-row hard cap, 1 MB/cell, 8 MB/result, blobs summarised as '<blob: N bytes>'. Streams rows so an oversized cell is caught after one fetch. New connection per call (no shared state).
- `app/tools/files.py` — read_file / list_dir: sandbox via Path.resolve() (follows symlinks and '..') + is_relative_to(data_dir) check; absolute paths allowed only if they resolve inside. 100 KB file cap, NUL-byte binary detection, UTF-8 only. Error messages echo the caller's path, never absolute host paths; results show sandbox-relative POSIX paths.
- `app/tools/seed.py` — Idempotent, race-safe seeding of demo.db (companies/vacancies/applications, ~25 rows). Threading lock + double-check + mkstemp temp file + os.replace atomic rename so concurrent first-run callers (threads or processes) never read a partial DB. CLI: `python -m app.tools.seed` / `make seed`. ensure_database() is also called lazily on every query_database call.
- `app/resources/docs.py` — Publishes data/docs files as concrete docs://<name> FunctionResources (md=>text/markdown, txt=>text/plain) plus a docs://{name} template. Every read delegates to files.read_file with docs_dir as sandbox root, so resources inherit the exact same escape/size/binary hardening as the read_file tool.
- `app/core/settings.py` — Frozen dataclass Settings resolved once from env: DATA_DIR (default = repo-root data/, computed as Path(__file__).parents[2]/'data'), MCP_HOST (0.0.0.0), MCP_PORT (8082). Derived paths: index_path=data/search_index.json, db_path=data/demo.db, docs_dir=data/docs. Injectable into create_server for tests.
- `app/core/logging.py` — Structured JSON logging: one JSON object per line to stderr (stdout carries stdio JSON-RPC frames). configure_logging is idempotent via module _CONFIGURED flag and clears root handlers. log_event(logger, msg, **context) attaches extras under record.context. Policy: log shapes/sizes/paths only — never SQL text, rows, or file contents.
- `app/core/errors.py` — ToolError(ValueError): the single tool-failure type whose message is safe to show callers; FastMCP converts it into a protocol-level tool error (isError=true, no traceback, no structuredContent).
- `tests` — 97 offline tests. conftest.py: index_path (the committed real index), sandbox (tmp data dir + a secret.txt OUTSIDE it), db_path (tmp, seeded on demand), rich_db_path (adversarial superset dataset — Cyrillic/CJK/emoji, ';' in data, salary 0 and 9_999_999 — built from seed._SCHEMA verbatim so it can't drift). test_mcp_integration.py runs a real in-memory MCP client via mcp.shared.memory.create_connected_server_and_client_session (skips, not fails, if the SDK moves that semi-private helper). test_concurrency.py fans out asyncio.to_thread calls to catch shared-state corruption and seeder races. test_settings.py is a regression guard for the data-dir default.
- `Dockerfile` — python:3.14-slim, pip-installs requirements.txt (NOT uv, NOT uv.lock), copies app/ and data/, runs as non-root appuser (uid 1000, data dir chowned so lazy seeding can write demo.db), EXPOSE 8082, TCP-connect healthcheck, CMD http transport. docker-compose.yml publishes 8082:8082 and bind-mounts ./data:/app/data.
- `.github/workflows` — ci.yml: uv sync --frozen; re-exports requirements*.txt and fails on git diff (lockfile/requirements drift gate); ruff on app+tests; strict mypy on app only; pytest; separate security job (pip-audit on requirements.txt + bandit on app). codeql.yml: CodeQL scanning. tag-release.yml: pushing a pyproject.toml version bump to main auto-tags v<version> and publishes a GitHub release whose notes are the matching '## [x.y.z]' CHANGELOG.md section verbatim — refuses to tag without that section, never re-tags an existing version.

## Invariants (breaking these breaks the platform)
- Logs MUST go to stderr: stdout carries the stdio transport's JSON-RPC frames, so any print()/logging to stdout corrupts the protocol. configure_logging installs a stderr handler; the one intentional stdout-free exception is the SSE-fallback notice in _run_http which explicitly targets sys.stderr.
- requirements.txt and requirements-dev.txt are GENERATED from uv.lock via `make lock` (uv export --frozen --no-hashes ...) — never hand-edit them. The Docker image installs from requirements.txt and never reads uv.lock, so drift means green tests but a container that fails at startup; CI's 'requirements.txt in sync with uv.lock' step enforces this with git diff --exit-code.
- data/demo.db is intentionally NOT committed (gitignored as data/demo.db and data/demo.db.*, also dockerignored): it is regenerated on demand by app/tools/seed.py, either explicitly or lazily on the first query_database call. Never commit it and never assume it exists on a fresh checkout.
- Tool modules under app/tools/ have ZERO mcp imports — they are pure typed functions taking explicit paths (db_path=, data_dir=, index_path=) as keyword-only args. All MCP coupling lives in app/server.py and app/resources/docs.py. Tests rely on this to call tools directly with tmp paths.
- Tool wrapper docstrings in app/server.py are the LLM-facing UI: FastMCP ships them as tool descriptions, and tests assert on their content (e.g. 'offline' must appear in search_web's description). The TypedDict return annotations generate the MCP output schemas — changing a TypedDict field changes the wire contract and test_tools_declare_output_schemas.
- Every tool failure must raise ToolError with a one-line, caller-safe message (no tracebacks, no absolute host paths); integration tests assert isError=true, structuredContent=None, and 'Traceback' absent from the text.
- The default DATA_DIR is the REPO-ROOT data/ directory, computed as Path(settings.py).parents[2]/'data'. Moving app/core/settings.py to a different depth silently breaks this; test_settings.py guards that the resolved default actually contains search_index.json and docs/.
- SQL safety comes from the sqlite3 authorizer + mode=ro URI, NOT from any string/regex filtering of the SQL text — do not 'improve' it by adding keyword blacklists. Authorizer denials are recognised by matching both SQLite phrasings: 'not authorized' and 'authorization denied'.
- Seeding must stay atomic and idempotent: unique mkstemp temp file + os.replace, guarded by _SEED_LOCK with a double-check. test_concurrency.py and test_database.py fire 20-25 parallel first-run calls and require every one to see a fully-seeded DB.
- The rich test DB fixture (tests/conftest.py) reuses seed._SCHEMA verbatim — schema changes in seed.py automatically propagate to the fixture; keep it that way instead of duplicating DDL.
- Tests must never write inside the repository: tests touching query_database pass a tmp_path-based data dir precisely so lazy seeding writes there, not into data/.
- Versioning is manifest-driven: the version string in pyproject.toml is the single source of truth; a release REQUIRES a matching '## [x.y.z]' section in CHANGELOG.md or tag-release.yml errors out.
- The project is not a package ([tool.uv] package = false); imports work because pytest.ini sets pythonpath=. and runtime uses `python -m app.server` from the repo root (or PYTHONPATH set explicitly, as the Claude Desktop config shows).

## Gotchas
- Importing app.server has side effects: module-level `server = create_server()` runs at import time — it configures root logging (clears existing root handlers) and lists data/docs to register resources, using REAL env settings (DATA_DIR etc.). Any `from app.server import create_server` triggers this. Construction does no file reads, but don't add heavier work to create_server without realizing it runs on import.
- The data-dir default already bit someone: it used to be app/data (containing only a stale demo.db), so bare `python -m app.server` broke search_web and registered zero docs:// resources while everything looked fine. The fix is _default_data_dir() -> repo-root data/; test_settings.py is the regression guard. If you relocate settings.py, parents[2] silently points somewhere else.
- requirements.txt drift is invisible locally: dev installs come from uv.lock, the Docker image from requirements.txt. Add a dependency without running `make lock` and tests stay green while the container crashes on import. Only the CI diff step catches it pre-merge.
- data/demo.db regenerates: it's gitignored and created lazily by the first query_database call in whatever DATA_DIR is active. Running the server from the repo root writes data/demo.db into the working tree (expected, ignored); with docker-compose the bind mount means the container writes demo.db onto the HOST ./data. Deleting it is safe; committing it is wrong.
- Docker runs as uid-1000 appuser and needs the data dir writable for lazy seeding — a bind-mounted ./data owned by a different uid on Linux hosts will make the first query_database fail with an OSError, while search_web and read_file still work.
- Claude Desktop sets no CWD for spawned servers: PYTHONPATH and DATA_DIR in claude_desktop_config.json must both be absolute paths, and `command` must be the venv interpreter if deps live in a venv.
- search_web is a stub, not a search engine: 14 curated entries, keyword-overlap scoring, reloaded from disk on every call. Tests assert exact ranking/tie-break determinism — reordering data/search_index.json entries changes tie-breaks and can break tests.
- Integration tests depend on two semi-private SDK surfaces: mcp.shared.memory.create_connected_server_and_client_session and FastMCP._mcp_server. Both are wrapped in skip-with-reason guards — after an mcp SDK upgrade, the whole integration suite may silently SKIP (not fail); check `pytest -rs` output, don't just trust green.
- The HTTP transport endpoint is /mcp (streamable HTTP), not /. The Docker healthcheck is a bare TCP connect (honouring MCP_PORT), so 'healthy' means the port answers, not that MCP handshakes succeed.
- mypy strict runs on app/ only — type errors in tests/ won't fail CI's typecheck step (ruff does cover tests).
- SQLITE_RECURSIVE may be missing from older sqlite3 builds; database.py falls back to the numeric value 33. Removing that getattr fallback breaks WITH RECURSIVE queries on some Pythons.
- max_rows/max_results are silently clamped (1..200 rows, 1..20 results), not rejected — callers asking for 10_000 rows get 200 with truncated=true and no error.
- Blobs are never returned: any bytes cell becomes the string '<blob: N bytes>'; a cell over 1 MB or a result over 8 MB raises ToolError mid-stream. Don't expect binary data through query_database.
- The umbrella-stack port for this service is 8082 (per the portfolio's port map in the user's memory notes); changing DEFAULT_PORT touches Dockerfile ENV/EXPOSE/healthcheck, docker-compose.yml, .env.example, README and the argparse help string — they are all hardcoded separately.

## Key flows
- stdio path: MCP client (Claude Desktop / Inspector) spawns `python -m app.server` -> JSON-RPC frames on stdout/stdin -> FastMCP dispatch -> thin wrapper in app/server.py logs a shape-only event -> pure function in app/tools/* -> TypedDict result returned as both JSON text and structuredContent; ToolError -> protocol tool error (isError=true).
- HTTP path: `python -m app.server --transport http` (or Docker CMD) -> streamable HTTP on MCP_HOST:MCP_PORT (default 0.0.0.0:8082, endpoint /mcp) -> same dispatch; falls back to legacy SSE transport if the installed mcp SDK predates streamable-http.
- query_database: sql string -> empty/length/multi-statement checks -> ensure_database(db_path) lazily seeds demo.db (lock + tempfile + atomic os.replace) -> fresh read-only (mode=ro) connection with authorizer -> stream rows with per-cell and total byte budgets -> {columns, rows, row_count, truncated}.
- search_web: query -> tokenize -> load data/search_index.json on every call (no cache) -> score = 2*keyword-overlap + 1*title-overlap -> sort by (-score, index position) -> top-N SearchHits with source='offline_index'.
- read_file/list_dir and docs:// resources: user path -> resolve() against sandbox root -> is_relative_to check -> read/list; docs.py routes resource reads through files.read_file rooted at data/docs, so docs://../demo.db is denied by the same sandbox.
- Release flow: bump version in pyproject.toml + add '## [x.y.z]' CHANGELOG section in a PR -> merge to main -> tag-release.yml greps the version, creates tag vX.Y.Z, publishes GitHub release with the changelog section as notes.
- Dependency flow: edit pyproject.toml -> `make lock` (uv lock + two uv exports) -> uv.lock drives dev installs, exported requirements.txt drives the Docker image, requirements-dev.txt drives nothing at runtime; CI fails if the exports drift from uv.lock.

## Conventions
- uv is the dependency source of truth: `make install-dev` = uv sync --frozen; regenerating lock + requirements exports is `make lock`. CI pins uv 0.11.7.
- Commands via Makefile: make run (alias for run-http), run-stdio, seed, test, lint (ruff check app tests), typecheck (mypy app — strict flags in pyproject, but only the app tree, not tests).
- Ruff line-length 100, target py310, rules E/F/I/W/B/UP with B008 ignored; pre-commit runs ruff --fix, mypy (local hook, pass_filenames=false), plus whitespace/yaml/toml/large-file hooks.
- requires-python >=3.10; typing uses `from __future__ import annotations`, TypedDict from typing_extensions (NotRequired support), TypeAlias for JsonScalar, frozen dataclasses for settings.
- Logging convention: log_event(log, 'message', key=value...) with non-sensitive scalar context only; logger names namespaced like 'mcp.tools'.
- Docstrings are load-bearing documentation everywhere — module docstrings explain design rationale (defence-in-depth ordering, why stderr, why the data-dir default changed); keep them updated when behavior changes.
- Tests: pytest with asyncio_mode=auto (async tests need no decorator), parametrised edge-case tests, comments stating the failure mode each concurrency test would catch.
- Structured tool results: every tool returns a TypedDict; blobs summarised, NotRequired fields omitted (not null) — test_list_dir_structured_content asserts size_bytes is absent for dirs.

## Test strategy
Fully offline, 97 tests, `make test` (uv run pytest; pytest.ini: testpaths=tests, pythonpath=., asyncio_mode=auto). Three tiers: (1) unit tests call the pure tool functions directly with injected tmp paths — test_search.py against the real committed index for determinism, test_database.py against a tmp seeded DB plus the adversarial rich_db_path fixture (unicode/emoji, embedded ';', salary edges 0 and 9_999_999), test_files.py against a tmp sandbox with a secret.txt planted OUTSIDE it and real symlink-escape cases; (2) test_concurrency.py drives asyncio.gather over asyncio.to_thread fan-outs (25 parallel queries, 20 parallel first-run seeds, mixed-tool interleaving) asserting exact results to catch shared-state or seeding races; (3) test_mcp_integration.py runs a real MCP client in memory via the SDK's create_connected_server_and_client_session against create_server(Settings(...)) — asserting tool listings, output schemas, structuredContent shapes, docs:// resources/mime types, and that ToolError maps to isError=true with no traceback; these skip with a reason if the SDK helper moves, so watch for silent skips after SDK bumps. CI = ruff (app+tests) + strict mypy (app only) + pytest + requirements-export drift check + pip-audit + bandit + CodeQL; pre-commit mirrors ruff/mypy locally. Convention: DB tests must pass tmp paths so lazy seeding never writes into the repo's data/.
